# 《云上小岛》完整闭环 Demo 开发手册

版本：玩法可行性验证版 v0.2 Draft

目标：先证明 2D 云岛放置经营的核心循环成立，再进入美术迭代。

## 1. 产品定位

《云上小岛》不是传统农场，而是一个轻量天气经营放置游戏。

玩家经营一个小型云岛，通过放置建筑、施放天气、升级设施和解锁地块，把手动点击逐步转化为半自动天气生产循环。

核心体验：

```text
我在岛上布置建筑 -> 我调度天气 -> 岛开始产出 -> 我升级和扩建 -> 天气链路越来越自动
```

## 2. 当前路线

当前采用：

- Vite
- React
- TypeScript
- HTML/CSS
- localStorage 后续补

当前不采用：

- Godot
- Phaser
- Pixi 复用
- 伪 2.5D 网格岛
- 图片资产优先路线

当前优先级：

```text
玩法闭环 > 信息清晰 > 手感反馈 > 数值节奏 > 美术资产
```

## 2.1 从 404 增量尝试迁移来的约束

404 的增量化尝试说明：如果一个游戏的核心乐趣来自空间解谜和结算解释，强行改成增量会很快变成手动推进、规则台账和 UI 面板堆叠。《云上小岛》应该避免这个问题，把重点放在自动增长和天气生产链上。

本项目后续增量设计必须满足：

- 不做手动推进天数 / 手动 tick。
- 资源和产出必须自动增长。
- 玩家操作用于加速、触发、收获、升级和解锁自动化。
- 解锁条件优先使用循环证明，例如收获、升级、自动湿润触发，而不是外部天数。
- 中央岛面必须表达成长、产出、天气影响和自动触发，侧栏只负责摘要和购买。
- 地块邻接只做轻量生产关系，不扩成空间优化谜题。

当前最重要的主目标：

```text
让玩家从手动下雨，进入第一条半自动天气生产链。
```

## 3. 核心循环

完整 Demo 必须跑通以下循环：

```text
放置建筑
-> 建筑每秒产出
-> 玩家施放天气影响局部地块
-> 作物成长 / 机器即时产出
-> 收获或资源增长
-> 升级建筑
-> 解锁新地块
-> 达成阶段目标
-> 解锁更稳定的天气循环
```

核心判断标准：

- 玩家知道下一步该做什么。
- 玩家每次点击地图都能看到反馈。
- 玩家能理解建筑和天气之间的关系。
- 玩家能在 10-15 分钟内搭出第一个半自动循环。
- 玩家能看到一个高层产出摘要，而不只是 6 个资源数字分别变化。

## 4. 核心资源

保留当前 6 类资源：

| 资源 | 作用 |
|---|---|
| 食物 | 麦田产出，早期目标和升级消耗 |
| 木头 | 伐木场产出，用于扩建和机器 |
| 石头 | 采石场产出，用于高级建筑 |
| 雨水 | 雨水收集器产出，用于天气链 |
| 阳光 | 阳光棱镜产出，用于成熟和高级升级 |
| 金币 | 通用建设资源 |

不新增资源，除非当前 6 类资源已经证明不够表达循环。

## 5. 建筑设计

保留当前 5 类建筑：

| 建筑 | 基础定位 | 天气关系 |
|---|---|---|
| 麦田 | 食物来源，唯一作物成长对象 | 下雨和晴天推动成长，成熟后收获 |
| 伐木场 | 木头来源 | 微风提供即时木头和金币 |
| 采石场 | 石头来源 | 微风提供即时石头和金币 |
| 雨水收集器 | 雨水来源 | 下雨提供即时雨水 |
| 阳光棱镜 | 阳光来源 | 晴天提供即时阳光 |

建筑字段建议：

```ts
interface BuildingDefinition {
  id: BuildingId;
  fullName: string;
  shortLabel: string;
  iconKey: string;
  icon: string;
  output: ResourceKey;
  baseRate: number;
  coinRate: number;
  cost: Partial<Record<ResourceKey, number>>;
}
```

## 6. 地块设计

地块状态：

- locked
- unlockable
- empty
- occupied
- selected
- mature
- weatherAffected

地块数据建议：

```ts
interface CellState {
  id: string;
  row: number;
  column: number;
  unlocked: boolean;
  buildingId?: BuildingId;
  level: number;
  growth: number;
}
```

地图仍使用 5x5。除非玩法验证证明空间太小，否则不要扩成更大地图。

## 7. 天气设计

保留三种天气：

| 天气 | 全局效果 | 局部施放效果 |
|---|---|---|
| 晴天 | 阳光、食物倍率提高 | 麦田成长，阳光棱镜即时产阳光 |
| 下雨 | 雨水、食物倍率提高，阳光降低 | 麦田成长，雨水收集器即时产雨水 |
| 微风 | 木头、石头、金币倍率提高 | 伐木场、采石场、金币即时收益 |

关键原则：

- 天气不是单纯倍率按钮。
- 天气必须能点到地图上，产生局部反馈。
- 天气施放要有范围、跳字、地块高亮。
- 全局倍率逻辑保留，局部施放增强互动感。

## 8. 阶段目标

阶段目标应从“数字目标”逐步转向“循环目标”。

阶段目标不要写成外部日程，例如“第 3 天解锁”。推荐写成玩家已经证明自己掌握了某条循环：

```text
收获 1 次麦田 -> 购买基础升级 -> 建造雨水收集器 -> 升到 Lv.2 -> 自动湿润相邻麦田 1 次
```

建议阶段：

### 阶段 1：岛开始产出

- 放置 3 座建筑。
- 收获 1 次成熟麦田。
- 食物达到 40。

### 阶段 2：天气开始参与

- 使用下雨推动麦田成长。
- 建造雨水收集器。
- 解锁 8 个地块。

### 阶段 3：形成第一条天气链

- 建造雨水收集器和阳光棱镜。
- 通过晴天或下雨获得即时资源。
- 建筑总等级达到 7。

### 阶段 4：进入半自动循环

- 建筑每秒总产出达到目标。
- 至少 3 个建筑等级达到 2。
- 玩家可以主要通过天气施放和升级维持生产。

v0.2 可以先做 3 阶段，第四阶段作为后续目标。

## 9. 自动化方向

完整闭环 Demo 需要一个很小的半自动目标，但不要做复杂 AI。

可选的最小自动化：

- 建筑升级到特定等级后，每隔一段时间自动触发一次对应天气收益。
- 雨水收集器达到 Lv.2 后，周期性湿润附近麦田。
- 阳光棱镜达到 Lv.2 后，周期性加速附近麦田。
- 伐木场 / 采石场在微风天气下额外自动跳一次收益。

第一版只需要一种自动化成立。

推荐：

```text
雨水收集器 Lv.2 -> 每 20 秒自动湿润相邻麦田一次
```

这样最贴合“从手动天气到半自动天气循环”。

## 10. UI 信息架构

保持当前三栏结构：

```text
左侧：资源、阶段目标、地块/解锁信息
中间：云岛地图、天气施放、反馈跳字
右侧：建筑购买、升级、删除、解锁等消耗项
```

原则：

- 资源和目标放左侧。
- 左侧需要一个总产出 / 秒、当前瓶颈和下一目标的高层摘要。
- 花资源的操作放右侧。
- 中央地图只显示游戏对象，不塞长文本。
- 地图格子保持 icon-first。
- 重要反馈必须出现在地图附近，不只改变左侧数字。
- 右侧购买列表优先突出“当前可买”和“最接近可买”，不要让玩家在很多按钮里找主线。

## 11. 反馈规范

每个核心动作至少有三层反馈：

| 动作 | 立即反馈 | 结果反馈 | 后续意义 |
|---|---|---|---|
| 放置建筑 | 格子变建筑 | 资源扣除 | 开始每秒产出 |
| 施放天气 | 鼠标/按钮进入施放态 | 地块高亮、跳字 | 成长或资源变化 |
| 麦田成熟 | 成熟标签/发光 | 可点击收获 | 获得食物并重置成长 |
| 升级建筑 | Lv 变化 | 产出提高 | 更快达成目标 |
| 解锁地块 | 新地块变空地 | 可放置空间增加 | 扩展布局 |

## 12. 最小状态增量

完整闭环 Demo 最多新增这些状态：

```ts
interface PrototypeState {
  activeWeatherTool: WeatherId | null;
  feedbackEvents: FeedbackEvent[];
  automationTimers?: Record<string, number>;
}
```

不做复杂状态机，不引入 Redux，不引入 ECS。

## 13. v0.2 实现批次

### Batch 1：闭环补强

- 阶段目标改为包含“收获麦田”和“使用天气”。
- 补资源不足 / 不可操作提示。
- 天气施放显示范围预览。
- 麦田成长和收获更稳定。

### Batch 2：第一条半自动链

- 雨水收集器 Lv.2 自动湿润相邻麦田。
- 自动触发时显示地图反馈。
- 阶段目标要求玩家完成该循环。

### Batch 3：节奏调优

- 调整成本、成长值、即时收益。
- 目标是 10-15 分钟内跑完第一条半自动链。
- 增加简单 debug summary，不做复杂 analytics。
- 对照手动试玩的 1 / 3 / 5 / 7 / 10 分钟节点，确认玩家每个阶段都有明确目标和产出变化。

### Batch 4：存档和交付

- localStorage 保存资源、地块、建筑、等级、成长、阶段。
- 增加 reset。
- 完成玩法可行性报告。

美术迭代从 Batch 4 之后开始。

## 14. 验收标准

完整闭环 Demo 必须满足：

- 可以放置 5 类建筑。
- 资源每秒增长。
- 可以升级建筑。
- 可以解锁相邻地块。
- 三种天气都能施放到地图上。
- 麦田可以成长、成熟、收获。
- 至少一条天气自动化链成立。
- 阶段目标能引导玩家体验该链。
- 10-15 分钟内能达到半自动循环。
- `npm run typecheck` 通过。
- `npm run build` 通过。
- 浏览器 console error 为 0。

## 15. 非目标

当前阶段不做：

- 新资源。
- 新建筑。
- 新作物系统。
- 剧情、NPC、任务文本。
- 多岛屿。
- 离线收益。
- 商店、仓库、每日奖励。
- 复杂天气预报。
- 拖拽摆放。
- 正式美术资产。
- image generation。
- 引擎迁移。

## 16. 下一轮推荐任务

下一轮直接实现 Batch 1：

```text
玩法闭环补强：
1. 阶段目标接入天气施放和麦田收获。
2. 补资源不足 / 不可操作提示。
```
