## 2026-05-15 试玩反馈记录：5 雨阶平台期与生产者链收益疑点

用户试玩进度：

- 玩到约 5 雨阶。
- 用时约 20 分钟。

反馈：

1. 5 雨阶后平台期开始明显出现。
   - 从 5 到 6、6 到 7 的雨阶需求跃迁都是 100 倍。
   - 同期升级成本也开始拉高，玩家需要等待一段时间。
   - 这可能说明雨阶需求表和生产者链解锁/收益曲线没有顺畅衔接。
   - 追加观察：严格来说平台期在雨阶 4 或雨阶 5 左右已经开始出现。
   - 触发点大约是天气活力达到 `1e8` 数量级后，所有升级都开始需要等待一段时间才能购买。
   - 同时雨阶重置需求开始以 100 倍速度扩大，升级等待和 reset 需求跃迁叠加，形成明显平台期。

2. 雨滴、根系、云团的链式加成感不明显。
   - 当前公式里生产者链是存量增强：`云团 -> 根系 -> 雨滴 -> 天气活力`。
   - 但玩家体感上每层对前一个资源的比例/存量加成不够明显，投资生产者链的价值感弱。

3. `根系苏醒` 文案和实际总收益变化存在严重预期落差。
   - 截图状态：`根系苏醒 Lv.9`，文案为“根系基础雨滴生成变为原来的 3 倍。”
   - 玩家观察：原先约 `20K/s`，购买后理论预期约 `60K/s`，实际约 `26K/s`。
   - 初步判断：这可能不是单纯显示 bug，而是“局部基础产出倍率”与“玩家理解的总 W/s 倍率”混在一起了。
   - 必须后续检查所有升级文案、公式和 UI 是否存在类似耦合/错读：
     - 局部基础倍率。
     - 当前资源存量贡献。
     - 生产者链层级贡献。
     - 最终天气活力 W/s。

4. `雨滴生成` 也存在同类问题，且玩家认为这是公式 BUG。
   - 截图状态：`雨滴生成 Lv.12`，文案为“天气活力基础增长变为原来的 4 倍。”
   - 玩家理解：天气活力增长应由“基础值 + 雨阶乘区 + 雨滴比例/存量加成 + 雨势增幅乘区”等组成。既然文案写“基础增长变为原来的 4 倍”，当前可见 W/s 就应该近似从 `1e10/s` 变为 `4e10/s`。
   - 玩家观察：购买后约为 `3.5e10/s`，没有达到 4 倍。
   - 玩家判断：这不是单纯文案歧义，而是升级公式或收益构成发生了非预期耦合。
   - 必须后续检查：
     - `dropletSeedRate + droplets` 是否导致 `dropletSeed` 只乘到生成器基础项，而没有乘到玩家理解的完整基础项。
     - `droplets` 存量是否稀释了 `dropletSeed` 升级带来的总收益倍数。
     - 如果升级卡写“变为原来的 N 倍”，是否必须保证当前可见 W/s 也接近 N 倍。
     - 如果设计上只能影响局部项，文案必须改成明确局部口径，并在 UI 另行显示预计 W/s 变化。

5. 后续公式方向倾向：让存量也吃到对应升级乘区。
   - 当前实现大致是：
     - `weather/s = (dropletSeedRate + droplets) * 后续乘区`
     - `droplets/s = (rootWakeBase + roots) * 后续乘区`
     - `roots/s = (cloudBloomBase + clouds) * 后续乘区`
   - 玩家预期的一般增量结构更接近：
     - 雨滴也应该吃到 `雨滴生成` 乘区，成为天气活力基础项的一部分；
     - 根系也应该吃到 `根系苏醒` 乘区，成为雨滴生产基础项的一部分；
     - 云团也应该吃到 `云团孕育` 乘区，成为根系生产基础项的一部分。
   - 另一种可选结构是把雨滴/根系/云团设计成乘区，而不是加法存量。
   - 当前用户倾向：先试“存量也吃到乘数”的版本。
   - 设计目标：升级卡写 `x4 / x3` 时，玩家看到的当前总收益应更接近同倍率变化，减少“买了但没变强”的错觉。

处理要求：

- 本条只记录，不改代码。
- 下一轮修复前必须先做公式审计，明确每个升级影响的是：
  - 本升级基础产出；
  - 当前资源每秒产出；
  - 下游资源每秒产出；
  - 最终天气活力 W/s。

## 2026-05-15 公式 A：存量吃对应乘区

本轮按试玩反馈改了公式，目标是让升级卡写 `x4 / x3` 时，玩家看到的当前收益变化更接近这个倍率。

改动范围：

- `weather/s`：雨滴存量现在也吃 `雨滴生成` 的 x4 乘区。
- `droplets/s`：根系存量现在也吃 `根系苏醒` 的 x3 乘区。
- `roots/s`：云团存量现在也吃 `云团孕育` 的 x3 乘区。
- 存档 key 改为 `cloud-island-weather-reactor-v12-lab-formula-a`，避免旧存档干扰本轮试玩。

公式方向：

```text
weather/s =
  (雨滴生成基础 + 雨滴存量)
  * 雨滴生成倍率
  * 雨阶倍率
  * 雨势增幅
  * 厚云降雨
  * 季风牵引
  * 其他乘区

droplets/s =
  (根系基础 + 根系存量)
  * 根系苏醒倍率
  * 生产者链乘区

roots/s =
  (云团基础 + 云团存量)
  * 云团孕育倍率
  * 生产者链乘区
```

验证时的模拟变化：

- 旧顺序脚本第一次季风从约 `128:23` 降到约 `19:18`。
- `producer-chain` 策略第一次季风约 `21:18`。
- 这说明“存量吃乘区”本身已经大幅打穿平台期，所以本轮没有继续降低雨阶需求表和升级成本。

风险：

- 这可能会让 `雨滴生成`、`雨势增幅` 的职责变得过近，二者都像外部乘区。
- 如果人类试玩也明显过快，下一轮要考虑：
  - 降低 `雨滴生成` 从 x4 到 x3；
  - 或把雨滴/根系/云团改为独立乘区资源；
  - 或恢复部分雨阶后段门槛。

## 2026-05-15 公式 A 试玩结论与下一版方向

用户试玩结论：

- formula A 确实让雨滴吃到 `雨滴生成` 加成后，需求平台出现骤降。
- 但不让雨滴吃加成时，又会太慢，且升级卡的 x4/x3 与实际体感不一致。
- 用户不倾向把 `雨滴生成` 从 x4 降到 x3，因为这会加大前期负担。

下一版倾向：

- 把雨滴从“基础加权项”中挪出来，改成独立乘区。
- 一个候选方向是 `log(雨滴)` 乘区，让雨滴存量有存在感，但不会因为吃完整 x4 而爆炸。
- 后续可以尝试类似：

```text
weather/s =
  dropletSeedBase
  * dropletMultiplier(droplets)
  * rainRankMultiplier
  * weatherAmplifierMultiplier
  * heavyRainMultiplier
  * monsoonPullMultiplier
  * otherMultipliers
```

候选口径：

```text
dropletMultiplier = 1 + log10(1 + droplets) * k
```

或：

```text
dropletMultiplier = (1 + log10(1 + droplets)) ^ p
```

待定参数：

- `k` 或 `p` 的大小。
- 雨滴乘区是否只影响被动天气活力，不影响点击。
- 根系/云团是否也使用类似 log 乘区，还是继续作为上游生产者。

核心设计问题：

- `雨滴生成` 和 `雨势增幅` 无论怎么调，都天然接近：一个是基础产出倍率，一个是外部天气乘区。
- 后续需要给它们重新分工：
  - `雨滴生成` 更偏“开局启动/基础产能”；
  - `雨势增幅` 更偏“全局乘区/阶段性爆发”；
  - 雨滴资源负责“随生产者链积累的额外乘区”。

## 2026-05-15 公式 B：雨滴改为 log 乘区，基础产能改为加法

本轮按用户最新方向建立 formula B，目标是保留雨滴资源的存在感，同时避免 formula A 中“雨滴存量吃完整 x4 后直接打穿平台”的问题。

存档 key：

```text
cloud-island-weather-reactor-v12-lab-formula-b
```

版本标识：

```text
Batch 1 / economy v12 lab formula B
```

公式方向：

```text
weather/s =
  活力基流(level)
  * 雨滴乘区(droplets)
  * 雨阶乘区
  * 天气增幅
  * 厚云降雨
  * 季风牵引
  * 其他乘区

活力基流(level) = level ^ 4

雨滴乘区(droplets) = 1 + log10(1 + droplets) * 50
```

本轮重命名：

| 旧名 | 新名 | 目的 |
|---|---|---|
| 云层触碰 | 云层注入 | 明确它影响点击注入天气活力 |
| 雨滴生成 | 活力基流 | 明确它是天气活力基础自动增长，不是雨滴资源 |
| 雨势增幅 | 天气增幅 | 明确它是天气活力外部乘区 |
| 雨滴余响 | 基流余响 | 和 `活力基流` 保持口径一致 |

当前职责拆分：

- `云层注入`：点击收益乘区。
- `活力基流`：天气活力被动增长的基础值，升级增加基础 W/s，当前为 `level^4`。
- `雨滴`：由生产者链积累，作为 log 乘区影响天气活力。
- `天气增幅`：天气活力外部乘区，仍然按 `x2/级`。

已知风险：

- `活力基流` 使用 `level^4` 是临时曲线，不保证最终平衡。
- `雨滴乘区` 当前只影响被动天气活力，不影响点击。
- 根系、云团仍沿用 formula A 的“存量吃对应生产升级乘区”，本轮没有继续改为 log 乘区。
- `天气增幅` 与雨滴 log 乘区都在最终天气活力上乘算，后续仍需要靠数值和解锁时机拉开职责。

## 2026-05-15 公式 C：基流线性 + 天气增幅提前

用户试玩反馈：前期压力仍偏大，且希望 `活力基流` 只承担基础值增长，主要加速交给乘区。

处理结果：

```text
活力基流(level) = level
每级升级只增加 +1/s 基础天气活力

天气增幅 = 4 ^ level
显示条件提前到云层等级 2，也就是 100 天气活力档
前期成本 = 100 / 1,000 / 10,000 / 30,000 / 100,000 / 300,000 / ...
```

活力基流成本改为前缓后陡：

```text
30 / 60 / 100 / 160 / 250 / 400 / 650 / 1,000 / 1,600 / 2,500 /
4,000 / 6,500 / 10,000 / 16,000 / 25,000 / 40,000 / 65,000 /
100,000 / ...
```

设计意图：

- `活力基流` 负责让玩家前期能多买几级基础值。
- `天气增幅` 负责把已有基础值放大，成为主乘区。
- 雨滴仍是生产者链积累后的 log 乘区。
- 纯 `点击 + 活力基流` 仍应卡住，避免单路线平推。

当前模拟观察：

- `early-click-auto`：第 1 雨阶 `8:54`，第 10 雨阶 `29:32`，第一次季风 `36:48`。
- `producer-chain`：第 1 雨阶 `8:54`，第 10 雨阶 `31:24`，第一次季风 `41:19`。
- `auto-rush`：第 1 雨阶 `101:53`，之后卡住，说明纯基流路线不可行。
- 当前顺序模拟第三次季风并唤醒天空心脏约 `55:34`。

风险：

- 这一版可能比 formula B 快很多，需要人类试玩确认是否太早进入 10 雨阶。
- `天气增幅 x4` 很强，后续可能需要调整后段成本，而不是再改公式。

## 2026-05-15 运行层修复：Formula C tick 与显示口径

用户确认当前跑的是 `FORMULA C`。本轮问题不是公式版本跑错，而是 `活力基流` 升级后增长观感异常。

排查结论：

- 纯经济函数测试中，购买 `活力基流 / 天气增幅 / 云层注入` 后，`runTick(state, 0.25)` 的实际增长都等于 `当前速率 * 0.25`。
- 公式层没有发现 `活力基流` 被额外重复乘算。
- 风险集中在 React 开发环境的 tick interval 重复执行。左侧资源卡片保持纯被动增长显示，点击收益只放在中间点击按钮上。

处理结果：

- `App.tsx` 的主循环改为基于 `performance.now()` 的真实时间 delta，不再硬编码每次 tick 都按 `0.25s`。
- 单次 tick 的 delta capped 到 `1s`，避免页面挂起或热更新后一次性补太多进度。
- 在开发环境中清理旧的全局 interval，降低 HMR / StrictMode 后多重 tick 的概率。
- 左侧 `天气活力` 的 `/秒` 显示保持纯被动收益，避免初始点击收益折算出默认 `+0.5/秒`。

已知口径限制：

- 点击收益当前只在中间云层按钮展示。
- 如果后续需要更严谨的调试 UI，可以额外拆出 `点击折算 / 秒`，但不要混进资源卡默认速率。

## 2026-05-15 显示与雨滴 log 乘区修正

用户反馈：

- 默认显示不应为了看完整整数被迫打开小数模式。
- 放置游戏不应在同一套基础显示里混用 `K/M` 和科学计数法。
- `1 + log10(1 + droplets) * 50` 在 `droplets = 1` 时已经约等于 `16.05x`，导致雨滴刚出现就击穿雨阶 2。

处理结果：

- 默认简洁显示下，`1,000` 到 `999,999` 之间的速率改为完整整数，不再显示 `K`。
- `1,000,000` 以上继续使用科学计数法，避免卡片溢出。
- 雨滴乘区改为：

```text
雨滴乘区(droplets) = 1 + log10(1 + droplets / 1,000) * 8
```

效果预期：

- `droplets = 1` 时乘区约 `1.003x`，不会秒推下一雨阶。
- `droplets = 1,000` 时乘区约 `3.41x`，雨滴开始有明确存在感。
- `droplets = 1,000,000` 时乘区约 `25x`，中后期仍然能承担生产者链的放大作用。

同时增加左侧本轮计时：

- 使用 `state.elapsedSeconds` 累计当前存档游玩时间。
- 暂停时计时停止。
- 重置存档后归零。
- 默认速率显示改为整数；`1,000,000` 以上统一进入科学计数法，不再混用 `K/M`。

### 2026-05-15 追加：实测增长读数

用户观察到 7 秒内天气活力增长远高于左侧被动速率。

追加 UI 读数：

- `被动 / 秒`：`calculateRates(state).weather`，只代表公式计算出的被动天气活力。
- `实测 / 秒`：用相邻两次状态的 `天气活力差值 / elapsedSeconds 差值` 得出，包含所有实际进入天气活力的来源。

用途：

- 如果 `实测 / 秒` 约等于 `被动 / 秒`，说明 tick 与公式一致。
- 如果 `实测 / 秒` 远高于 `被动 / 秒`，说明有点击、自动点击、旧存档残留或其他未计入被动面板的来源正在注入天气活力。

## 2026-05-15 生产者链收束：根系 / 云团改为 log 存量

用户试玩观察：

- 约 `29min` 到达 `8` 雨阶，说明雨阶 6 后仍然过快。
- 当前生产者链使用 `(基础 + 存量) * 升级倍率`，会让倍率升级同时放大已经积累的根系和云团存量。
- 雨滴已经改为 log 乘区后，根系和云团继续线性加权会在中段把雨滴乘区再次推爆。

处理结果：

```text
雨滴生成 = 根系基础生成器 * 根系升级倍率 * 根系存量log乘区 * 生产者公共乘区
根系生成 = 云团基础生成器 * 云团升级倍率 * 云团存量log乘区 * 生产者公共乘区

生产者存量log乘区(amount) = 1 + log10(1 + amount / 1,000) * 6
```

设计口径：

- `根系苏醒` 和 `云团孕育` 只放大对应基础生成器。
- 根系 / 云团存量不再线性进入下游基础值，而是通过 log 变成缓慢增长的存量回报。
- 这保留了“生产者链越积越有用”的感觉，但避免一买倍率就把旧存量整体炸穿。

雨阶 6 后需求同步恢复更强坡度：

```text
... / 1e13 / 1e15 / 1e17 / 1e19 / 1e20 / 1e21 / ...
```

目的：

- 让雨阶 6 以后重新形成平台期。
- 防止 6 -> 8 雨阶在生产者链解锁后被几分钟平推。

当前模拟观察：

- `producer-chain`：第 6 雨阶约 `24:11`，第 8 雨阶约 `28:51`，第 10 雨阶约 `34:56`。
- `early-click-auto`：第 6 雨阶约 `23:44`，第 8 雨阶约 `27:55`，第 10 雨阶约 `33:54`。
- `value-greedy` 仍偏快，第 10 雨阶约 `29:07`，说明贪心脚本会利用当前成本结构压缩平台期，不一定代表人类路线。

风险：

- 这次已经修掉“旧存量被升级倍率整体放大”的公式问题。
- 但模拟显示 6 -> 10 雨阶仍可能偏快，后续如果人类试玩也偏快，应优先调 `rootWake / cloudBloom / windEye` 成本曲线或生产者 log 系数，而不是再放宽雨阶需求。
