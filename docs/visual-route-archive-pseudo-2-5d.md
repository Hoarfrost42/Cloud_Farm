# 《云上小岛》伪 2.5D 路线归档

## 1. 归档结论

伪 2.5D / 像素网格岛路线已归档，当前不再作为《云上小岛》后续主路线。

这条路线包括：

- 像素风伪 2.5D 浮岛资产。
- 岛面投影坐标系。
- 岛面网格化摆放。
- Phaser 轻量 5x4 网格岛实验。
- 以类似经营手游主界面的 2.5D 网格岛为参考方向。

归档原因不是玩法失败，而是表现路线对 v0.1 造成了过大的架构和资产负担。

## 2. 已探索内容

已完成过的探索：

- Art Pass 0：伪 2.5D 美术验证计划。
- Art Pass 1：三张方向图，包含 pixel / clean 2D / hybrid chunky。
- Art Pass 2：转向 asset-first，并尝试生成像素资产板。
- Pixi 场景内放大浮岛和接入像素 PNG 资产。
- `src/game/sceneLayout.ts` 中建立过岛面连续投影坐标。
- 基于用户草图讨论过岛面格位与三块地块摆放。
- 新增过 Phaser 实验场景：
  - `src/phaser/PhaserGameCanvas.tsx`
  - `src/phaser/scenes/CloudIslandScene.ts`

## 3. 主要问题

### 3.1 坐标系统成本上升

原本 v0.1 只需要普通 2D 坐标。伪 2.5D 引入后，需要处理：

- 岛面投影。
- 网格格位。
- 前后层级。
- 对象占格。
- 地块姿态。
- 机器摆放避让。

这些工作已经接近一个小型场景布局系统。

### 3.2 资产规范要求过高

伪 2.5D 要求所有资产角度一致：

- 岛面。
- 地块。
- 作物阶段。
- 机器底座。
- 云和特效层级。

只要某个对象角度不对，就会像贴纸漂在岛上。

### 3.3 项目焦点偏离

《云上小岛》的核心目标是轻松放置循环：

```text
点云 -> 下雨 -> 作物成长 -> 收获 -> 升级 -> 机器自动化
```

伪 2.5D 路线让大量时间被消耗在视觉架构、格子投影和资产拼接上，削弱了 v0.1 应该优先验证的轻松反馈。

### 3.4 Phaser 实验的额外成本

Phaser 能做 2D 游戏场景，但当前实验带来了：

- 额外引擎依赖。
- bundle 体积显著增加。
- 与 React 宿主布局的集成问题。
- 与既有 Pixi 文件并存的维护成本。

当前不继续沿 Phaser 伪 2.5D 网格岛路线推进。

## 4. 保留价值

这条路线仍有可复用经验：

- 中央舞台应该更像游戏画面，而不是 Web dashboard。
- 地块和机器需要有清晰的空间关系。
- 右侧 UI 应该更像游戏 HUD，少一些后台表单感。
- 天气主题需要比普通农场更突出。
- 资产应优先服务核心循环，而不是追求大场景完整度。

## 5. 后续处理建议

当前先不急着删除代码。

下一轮如果开始实现 2D 新路线，应先决定：

1. 是否保留 Phaser 作为 2D 渲染引擎。
2. 是否回退到 PixiJS。
3. 是否清理 `src/phaser/` 和 `phaser` 依赖。
4. 哪些 PNG 资产可作为临时占位继续使用。

在新路线完成最小可玩验证前，不建议继续在 2.5D 网格岛上修补。
