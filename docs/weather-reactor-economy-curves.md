## 核心公式

### 天气活力每秒增长

```text
baseWeatherPower = dropletSeedWeatherRate
dropletMultiplier = 1 + log10(1 + droplets) * 50
rainRankMultiplier = 1 + rainRanks
weatherAmplifierMultiplier = 4 ^ weatherAmplifier
heavyRainMultiplier = 3 ^ heavyRain
stormMultiplier = 1 + stormMemory * max(1, monsoonCycles) * 0.12
coreMultiplier = 1 + totalCloudCores * 0.03

passiveWeather =
  baseWeatherPower
  * dropletMultiplier
  * rainRankMultiplier
  * weatherAmplifierMultiplier
  * heavyRainMultiplier
  * stormMultiplier
  * coreMultiplier

autoDrizzleWeather =
  autoDrizzle > 0 ? cloudTouchAmount * autoDrizzle * 0.07 : 0

weatherPerSecond = passiveWeather + autoDrizzleWeather
```

### 活力基流

```text
dropletSeedRate(level) =
  level > 0 ? level : 0
```

### 雨滴乘区

```text
dropletMultiplier(droplets) =
  1 + log10(1 + droplets) * 50
```

### 生产者链

```text
dropletsPerSecond =
  rootWake > 0
    ? (1 * 3 ^ (rootWake - 1) + roots) * livingSoilMultiplier
    : 0

rootsPerSecond =
  cloudBloom > 0
    ? 1 * 3 ^ (cloudBloom - 1) + clouds
    : 0

cloudsPerSecond =
  windEye > 0
    ? 1 * 2 ^ (windEye - 1)
    : 0
```

`livingSoilMultiplier` 是云核永久升级，拥有后为 `1.25`，否则为 `1`。

### 点击收益

```text
cloudTouchAmount =
  1
  * 5 ^ cloudTouch
  * (1 + rainRanks)
  * 3 ^ heavyRain
```

点击收益不再吃冷却时间，也不再使用已经删除的点击翻倍升级。

## 小数和耦合审计

### 已修复：升级文案倍率耦合

之前 `雨滴生成 Lv.7` 会显示 `3.946 倍`，原因是文案拿了：

```text
购买后全局天气活力每秒 / 购买前全局天气活力每秒
```

这个全局比值会被当前雨滴存量、雨阶、雨势增幅、厚云降雨、云核乘区影响，所以不会等于升级自己的 x4。

formula C 使用局部描述：

```text
活力基流：天气活力基础增长增加 N/秒
云层注入：点击注入变为原来的 5 倍
天气增幅：天气活力外部乘区变为原来的 4 倍
```

### 仍然存在的小数来源

这些是当前公式真实会产生小数的地方：

| 来源 | 原因 |
|---|---|
| tick 间隔 | 游戏每 `0.25s` 结算一次，资源会出现小数累积 |
| 活土 | `livingSoilMultiplier = 1.25` |
| 雨滴 log 乘区 | `log10(1 + droplets) * 50` |
| 自动细雨 | `cloudTouchAmount * level * 0.07` |
| 风暴记忆 | 每级每季风 `0.12` 加成 |
| 云核总数 | 每个总云核 `0.03` 加成 |
| JavaScript 浮点 | `0.1/0.03/0.055` 这类十进制不能被二进制浮点精确表示 |

### 展示策略

当前普通显示：

- 小数会被四舍五入成整数或 0.5 步进。
- `>= 1e6` 的数字使用科学计数法。
- 不再使用 `M/B/T` 或更高字母后缀。

当前精确显示：

- 小于 `1e6` 时最多显示 3 位小数。
- 大于等于 `1e6` 时使用 6 位有效小数的科学计数法。

## 当前判断

formula C 把 `活力基流` 和 `天气增幅` 拆开：前者只加基础值，后者承担主要乘区。当前自动脚本显示首通已经明显压缩，下一轮需要用人类试玩确认是否过快、以及生产者链是否仍然有足够存在感。
