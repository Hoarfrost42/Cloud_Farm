## 4. 公式演化记录

### 4.1 Formula A：存量吃对应乘区

目的：

- 修复“升级写 x4/x3，但玩家实际收益没接近 x4/x3”的体感问题。

做法：

```text
weather/s = (活力基流 + 雨滴存量) * 雨滴生成倍率 * 后续乘区
droplets/s = (根系基础 + 根系存量) * 根系倍率
roots/s = (云团基础 + 云团存量) * 云团倍率
```

结论：

- 升级体感变直接。
- 平台期被大幅击穿。
- 生产者存量被升级倍率整体放大，导致增长过爆。

### 4.2 Formula B：雨滴改为 log 乘区

目的：

- 保留雨滴存量价值，但避免雨滴吃完整 x4 后爆炸。

做法：

```text
weather/s =
  活力基流(level)
  * 雨滴log乘区
  * 雨阶乘区
  * 天气增幅
  * 其他乘区
```

早期参数曾为：

```text
雨滴乘区 = 1 + log10(1 + droplets) * 50
```

结论：

- 方向正确。
- 参数过猛，`droplets = 1` 时就约 `16.05x`，直接击穿前期曲线。

### 4.3 Formula C：当前版本

Formula C 的核心分工：

| 机制 | 当前职责 |
|---|---|
| 云层注入 | 点击收益乘区 |
| 活力基流 | 天气活力基础自动增长，线性加法 |
| 天气增幅 | 天气活力外部乘区 |
| 雨滴 | 生产者链积累出的天气活力 log 乘区 |
| 根系 / 云团 | 生产者链存量 log 乘区 |
| 雨阶 | 小 reset 和天气活力全局乘区 |
| 季风牵引 | 10 雨阶后的季风冲刺乘区 |

## 5. 当前核心公式

### 5.1 天气活力被动增长

```text
weather/s =
  活力基流
  * 雨滴log乘区
  * 雨阶乘区
  * 天气增幅乘区
  * 厚云降雨乘区
  * 季风牵引乘区
  * 风暴记忆乘区
  * 云核乘区
  + 自动细雨
```

代码口径：

```text
活力基流(level) = level

雨滴log乘区 =
  1 + log10(1 + droplets / 1,000) * 8

雨阶乘区 =
  1 + rainRanks

天气增幅乘区 =
  4 ^ weatherAmplifier

厚云降雨乘区 =
  3 ^ heavyRain

季风牵引乘区 =
  100 ^ monsoonPull

风暴记忆乘区 =
  1 + stormMemory * max(1, monsoonCycles) * 0.12

云核乘区 =
  1 + totalCloudCores * 0.03
```

雨滴 log 当前点位：

| 雨滴 | 乘区 |
|---:|---:|
| 0 | `1.00x` |
| 1 | `1.003x` |
| 1,000 | `3.41x` |
| 1,000,000 | `25.00x` |

### 5.2 生产者链

当前链条：

```text
风眼 -> 云团 -> 根系 -> 雨滴 -> 天气活力
```

当前生产者存量 log：

```text
生产者存量log乘区(amount) =
  1 + log10(1 + amount / 1,000) * 5
```

当前公式：

```text
droplets/s =
  1
  * 3 ^ (rootWake - 1)
  * producerStockMultiplier(roots)
  * producerCommonMultiplier

roots/s =
  1
  * 3 ^ (cloudBloom - 1)
  * producerStockMultiplier(clouds)
  * producerCommonMultiplier

clouds/s =
  1
  * 2 ^ (windEye - 1)
  * producerCommonMultiplier
```

公共生产者乘区：

```text
producerCommonMultiplier =
  livingSoilMultiplier
  * (1 + monsoonFocus * 0.5)
```

其中 `livingSoilMultiplier` 拥有活土后为 `1.25`，否则为 `1`。

## 8. Reset 规则

### 8.1 雨阶 Reset

条件：

```text
weather >= getRainRankRequirement(state)
```

效果：

- `rainRanks += 1`
- 清空本轮资源。
- 清空本轮升级。
- 保留云核、总云核、永久升级、季风循环次数、天空心脏状态、计时。
- 保留 `monsoonPull`，因为它属于更高一级季风冲刺。
- 如果拥有 `凝雨熟练`，雨阶 reset 后保留第一组升级：
  - 云层注入
  - 活力基流
  - 天气增幅

已修问题：

- 早期雨阶 reset 没清右侧升级，导致快速连凝雨阶。已修。
- 早期雨阶可能批量获得多个雨阶，UI 未提示。已改成一次只获得 1 雨阶。

### 8.2 季风 Reset

条件：

```text
weather >= monsoonTarget
rainRanks >= 10
windEye > 0
```

第一次季风目标：

```text
1e20
```

季风目标增长：

```text
每次季风后 x10
```

效果：

- `monsoonCycles += 1`
- 获得云核。
- 清空本轮资源和本轮升级。
- 清空 `monsoonPull`。
- 保留永久升级。

云核收益：

```text
baseCoreGain = 3 + floor((weather / target) ^ 0.25)
rankBonus = floor(max(0, rainRanks - 10) / 2)
lensBonus = hasMonsoonLens ? 1 : 0
cloudCoreGain = clamp(baseCoreGain + rankBonus + lensBonus, 4, 12)
```

当前第一次季风至少给 `4` 云核。

### 10.2 本轮计时

左侧增加本轮计时器：

- 使用 `state.elapsedSeconds`。
- 暂停时停止。
- 重置存档后归零。
- 用于记录每个阶段真实用时。

### 10.3 被动 / 实测读数

计时器下方增加：

```text
被动 x/s · 实测 y/s
```

含义：

- `被动`：`calculateRates(state).weather`。
- `实测`：相邻状态的天气活力差值 / 经过时间。

用途：

- 判断实际增长是否和公式面板一致。
- 曾经观测到 7 秒内增长约 10 倍于面板速率，追加该读数后确认当前状态已经对齐。

### 10.4 暂停

左侧增加暂停按钮：

- 暂停 tick。
- 计时停止。
- 方便观察阶段数据。

## 11. 运行层修复

### 11.1 Tick 改真实 delta

早期 tick 固定每次按 `0.25s` 结算。v12 修为：

```text
elapsedSeconds = (performance.now() - previous) / 1000
```

并限制：

```text
singleTickDelta <= 1s
```

目的：

- 避免页面挂起或热更新后一次性补过多。
- 让计时和资源增长更一致。

### 11.2 开发环境 interval 清理

在 React StrictMode / Vite HMR 下，旧 interval 可能残留。

处理：

- 在 `window` 上记录当前 weather tick interval。
- 新 interval 启动前清理旧 interval。
- cleanup 时只清当前匹配 interval。

当前结论：

- 最新状态下 `被动/s` 与 `实测/s` 已对齐。
- 如果后续再次拉开，优先截图这两个读数。

### 11.3 季风 reset 保留计时

早期 `runMonsoonCycle()` 使用 `createAppState()` 开新轮时没有传入 `elapsedSeconds`，导致第一次季风后左侧计时归零。

当前处理：

- 季风 reset 保留 `elapsedSeconds`。
- 重置存档时仍归零。
- 这样后续记录第一次季风、第二次季风、风暴前线与气候改写的累计时间不会被季风 reset 打断。

1. `6 -> 10` 雨阶仍可能偏快。
   - 短线 ROI 仍可在约 29 分钟达成 10 雨阶。
   - 如果目标是几小时小品，需要后续拉长季风前后，而不是只拉 10 雨阶前。

2. 生产者链职责还需要更清晰。
   - 雨滴 log、根系 log、云团 log 已经缓和爆炸。
   - 但每层升级的“买了以后变强多少”仍需要更明确 UI 反馈。

3. 季风后的自动化还没有充分试玩。
   - 自动细雨、自动雨阶、雨阶压缩、季风聚焦、风暴记忆都有定义。
   - 但季风后 2-4 轮曲线还未细调。

4. 数字动画和游戏感还弱。
   - 当前主要是 React 文本刷新。
   - 还没有经典增量游戏那种高速滚动、增量浮字、reset 爆发反馈。

5. 仍使用 JavaScript `number`。
   - 当前数值范围尚可。
   - v13 已开始准备 `logNumbers.ts`，但当前 v12 tick 尚未接入 log-safe 结算。

## 15. 下一轮建议

建议暂停结构重构，进入小步调参。

优先顺序：

1. 试玩完整第一季风。
2. 记录：
   - 第 1 / 3 / 6 / 8 / 10 雨阶时间。
   - 第一次季风时间。
   - 第一次季风给多少云核。
   - 云核第一笔升级是否立刻压缩旧流程。
3. 如果 10 雨阶仍太快：
   - 优先提高 `rootWake / cloudBloom / windEye` 后段成本；
   - 或继续降低 `PRODUCER_STOCK_LOG_MULTIPLIER_STEP`；
   - 或继续提高雨阶 8-10 需求。
4. 如果季风太远：
   - 优先调 `monsoonPull` 成本和 `BASE_MONSOON_WEATHER_TARGET`；
   - 不要再加强雨阶倍率。
5. UI 后续增加买前买后预览：
   - 当前被动/s。
   - 购买后被动/s。
   - 回本时间。
