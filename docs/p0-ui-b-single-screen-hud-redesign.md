# P0-UI-B 单屏空岛 HUD 重构方案

适用项目：Cloud Island / 云上小岛 / Weather Reactor  
适用版本：v13 / Post-Monsoon Complete Slice  
当前分支背景：`p0-h-balance-ver-b`  
文档目的：把当前三栏信息面板重构方向整理为可评审方案，交给 GPT Pro / 后续视觉原型继续讨论。

## 0. 当前结论

当前 UI 最大问题不是某张卡片不好看，而是信息架构仍然像调试面板：

```text
左栏：资源与计时
中栏：云层舞台 + 生产者链 + 多层 reset
右栏：升级与后期系统
```

这个结构开发方便，但玩家体验上会产生四个问题：

1. 开局暴露太多未理解资源。
2. 中央舞台被 reset 卡片和固定高度挤压。
3. 侧栏和中心区域需要滚动才能看完。
4. Mood / 空岛生态没有成为主视觉，玩家主要在读数字和卡片。

本方案建议把 UI 改成单屏可读的游戏 HUD：

```text
顶部：紧凑 HUD，显示当前主资源、阶段、目标、运行状态
中央：空岛天气舞台，承担 Mood、阶段反馈、主要点击对象
底部：当前行动条，显示当前最重要的升级 / reset / 下一步
二级入口：资源、升级全览、公式、记录、设置，以弹层或抽屉打开
```

核心约束：

```text
默认首屏不出现滚动条。
开局不暴露后期资源。
同一时刻只突出当前阶段最重要的 1-3 个动作。
Mood 只改变视觉和文案，不写存档、不进公式、不影响模拟。
公式、完整资源、调试信息默认退到二级入口。
```

## 1. 为什么当前三栏结构不适合

### 1.1 三栏结构的来源

三栏是很常见的开发默认结构：

```text
状态栏 / 主舞台 / 操作面板
```

它适合后台、仪表盘、策略游戏编辑器、调试工具。增量游戏也常见这种结构，因为资源、升级、目标都可以直接平铺。

但 Cloud Island 的核心体验是：

```text
玩家推动天气，空岛逐渐变得有生命。
```

如果 UI 以三栏资源表为中心，玩家会先感受到“我在看很多数字”，再理解“空岛发生了什么”。这会削弱游戏主题。

### 1.2 当前三栏造成的具体问题

| 问题 | 表现 | 风险 |
|---|---|---|
| 开局资源过载 | 云核、季风、气压、风暴胞、气候织线提前出现 | 玩家以为自己漏了系统 |
| reset 卡片堆叠 | 中栏底部出现不可见或被裁切卡片 | 玩家不知道当前能做什么 |
| 滚动条依赖 | 左栏和中栏都可能需要滚动 | 单屏游戏变成后台面板 |
| 操作焦点分散 | 点击云层、买升级、看目标分处三列 | 视线来回跳 |
| Mood 弱化 | 背景和天气效果只是装饰 | 进度缺少情绪反馈 |

### 1.3 当前过渡实验的价值与局限

已经完成的右侧“升级工作台 + 页签”实验有价值：

- 证明后期系统可以从首屏移走。
- 证明页签能降低右侧堆叠。
- 保留了当前代码路径，风险较低。

局限也明显：

- 仍然保留三栏。
- 左侧资源过载未解决。
- 中央 reset 卡片仍可能溢出。
- 首屏仍然是面板优先，舞台感不足。

因此它适合作为参考，不应作为最终 UI 方向。

## 2. 目标体验

玩家在任意阶段打开游戏，应该在 3 秒内理解：

```text
我现在处在哪个天气阶段。
我当前最重要的资源是什么。
我下一步该做什么。
我做完以后会进入什么变化。
空岛因为我的进度发生了什么视觉改变。
```

玩家不需要在开局理解：

```text
云核是什么。
气压是什么。
风暴胞是什么。
气候织线是什么。
天空心脏如何结算。
完整公式如何拆分。
```

这些内容应按阶段逐步显露。

## 3. 新布局草案

### 3.1 桌面端基础布局

推荐桌面结构：

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶部 HUD：天气活力 / 当前阶段 / 下一目标 / 运行状态 / 菜单入口 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                     中央空岛天气舞台                         │
│      云层、雨线、根系、风线、风暴边缘、气候环、天空心脏       │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│ 底部行动条：当前推荐升级 1-3 个 + 当前 reset / 主线动作       │
└─────────────────────────────────────────────────────────────┘
```

二级入口可以放在右上角或顶部 HUD 末端：

```text
资源详情
升级全览
天气图谱
公式
记录
设置
```

这些入口打开后使用弹层、抽屉或全屏 overlay，不常驻首屏。

### 3.2 视觉层级

优先级从高到低：

1. 中央空岛 Mood 和当前阶段。
2. 当前主资源与下一目标。
3. 当前可执行动作。
4. 当前可购买升级。
5. 资源详情与公式。
6. 后期系统预告。
7. 调试信息。

### 3.3 不使用常驻滚动条

本项目首屏应尽量无滚动条。原因：

- idle 游戏允许玩家短暂查看状态，不应像管理后台一样扫长列表。
- 后期系统多时，滚动只会隐藏问题。
- 关键行动被滚动隐藏会造成失败体验。

允许使用滚动的地方：

```text
弹层内部的完整升级列表
公式详情
调试记录
长文本帮助
```

不允许使用滚动的地方：

```text
主舞台
顶部 HUD
底部行动条
当前关键 reset
当前推荐升级
```

## 4. 阶段化信息显露

### 4.1 资源显示规则

资源应按实际进入阶段显露。

| 阶段 | 顶部 HUD 显示 | 详情中可见 | 不显示 |
|---|---|---|---|
| 开局 | 天气活力、雨阶 | 天气活力、雨阶 | 云核、气压、风暴胞、气候织线 |
| 雨阶 1-2 | 天气活力、雨阶、雨滴 | 雨滴、被动/s、点击收益 | 云核、气压、风暴胞、气候织线 |
| 雨阶 3-5 | 天气活力、雨阶、雨滴、根系 | 生产者链详情 | 云核、气压、风暴胞、气候织线 |
| 雨阶 6-10 | 天气活力、雨阶、风眼进度 | 雨滴、根系、云团、风眼 | 云核、气压、风暴胞、气候织线 |
| 第一次季风后 | 天气活力、雨阶、季风、云核 | 云核天赋、气压 | 风暴胞、气候织线 |
| 风暴前线后 | 天气活力、当前前线、风暴胞 | 风暴图谱 | 气候织线 |
| 气候改写后 | 天气活力、气候织线、当前法则 | 气候法则 | 终局细节按需要显露 |
| 天空心脏阶段 | 天气活力、天空心脏脉冲 | 终局目标、全部层级 | 无 |

左侧当前那种资源网格应改成“阶段摘要”，不再开局显示所有槽位。

### 4.2 主目标显示规则

顶部 HUD 只显示一个主目标：

```text
当前主线：第一次季风
目标：天气活力达到 1e20，雨阶达到 10
进度：orders 或百分比
```

详细条件进入目标详情弹层。

如果有可执行 reset：

```text
主目标位置切换成“可执行动作”
底部行动条显示 reset 按钮
```

### 4.3 升级显示规则

首屏只显示推荐升级，不显示完整升级树。

推荐升级数量：

```text
开局：1-2 个
雨阶中段：2-3 个
季风后：2-3 个
风暴后：2-4 个，但要按主干 / 分支区分
```

完整升级树进入“升级全览”弹层。

弹层内可以有页签：

```text
本轮升级
云核天赋
气压升级
风暴图谱
气候法则
```

但未解锁页签不显示，或只在“即将解锁”状态下出现。

## 5. Mood 系统作为主视觉

### 5.1 Mood 定位

Mood 是纯表现层：

```text
不写入存档。
不改变数值。
不影响模拟器。
不参与购买条件。
只从 WeatherReactorState 派生。
```

Mood 的职责：

```text
告诉玩家空岛正处于什么状态。
让阶段推进可感知。
减少纯数字等待的无聊感。
给每次大 reset 后一个视觉反馈。
```

### 5.2 Mood 阶段草案

| Mood | 触发建议 | 视觉关键词 | 玩家感受 |
|---|---|---|---|
| `dryStart` | `rainRanks === 0 && totalMonsoonCycles === 0` | 干净天空、轻云、空旷 | 刚开始唤醒天气 |
| `firstRain` | `rainRanks >= 1 && rainRanks < 3 && totalMonsoonCycles === 0` | 细雨、浅蓝水汽 | 第一次雨来了 |
| `rootedRain` | `rainRanks >= 3 || rootWake/cloudBloom > 0` 且未季风 | 根系、嫩绿、稳定雨滴 | 生态开始扎根 |
| `windEye` | `rainRanks >= 6 || windEye > 0` 且未季风 | 风线、云层流动 | 天气开始循环 |
| `monsoon` | `totalMonsoonCycles >= 1 && totalStormFronts === 0` | 厚云、湿润天空、云核辉光 | 进入新循环 |
| `stormFront` | `totalStormFronts >= 1 || milestone.kind === "stormFront"` | 斜雨、暗云边缘、闪光 | 前线形成 |
| `climateRewrite` | `totalClimateRewrites >= 1 || milestone.kind === "climateRewrite"` | 气候环、层状纹理、稳定光带 | 规则被改写 |
| `skyHeart` | `skyHeartPulseLevel > 0 || bestWeatherExp >= 250 || skyHeartAwakened` | 高空辉光、脉冲、终局光晕 | 接近天空心脏 |

注意：

```text
不要用 pressure > 0 直接触发 stormFront。
第一次季风后就可能有气压，过早触发风暴视觉会误导玩家。
```

### 5.3 Mood 应影响的 UI 区域

Mood 可以影响：

- 背景天空色。
- 云层密度和形状。
- 雨线 / 风线 / 气候环等 CSS 伪元素。
- 中央 sky-status 文案。
- 生产者链的活跃形态。
- 大 reset 后的短暂反馈。

Mood 不应影响：

- 资源计算。
- 购买条件。
- 模拟器。
- 存档结构。
- 按钮可读性。

## 6. 推荐 UI 组件拆分

当前 `src/App.tsx` 同时承担状态、handler、布局、卡片渲染、文案格式化，后续 UI 重构会越来越难。

建议按 P0 最小拆分，不拆状态管理：

```text
src/ui/TopHud.tsx
src/ui/IslandMoodStage.tsx
src/ui/ActionBar.tsx
src/ui/UpgradeOverlay.tsx
src/ui/ResourceOverlay.tsx
src/ui/FormulaDrawer.tsx
```

P0 只拆 UI 展示，不抽全局状态。

### 6.1 `TopHud`

职责：

```text
显示当前阶段、主资源、下一目标、运行状态。
只展示当前阶段资源。
提供二级入口按钮。
```

输入建议：

```ts
interface TopHudProps {
  state: WeatherReactorState;
  nextGoal: GoalViewModel;
  mood: IslandMood;
  isPaused: boolean;
  onTogglePause: () => void;
  onOpenOverlay: (overlay: OverlayId) => void;
}
```

### 6.2 `IslandMoodStage`

职责：

```text
承载中央舞台。
显示云层按钮、生产者链、Mood 天气效果。
不显示完整资源列表。
不显示后期系统列表。
```

输入建议：

```ts
interface IslandMoodStageProps {
  state: WeatherReactorState;
  mood: IslandMood;
  canTouchCloud: boolean;
  cloudCooldownProgress: number;
  onTouchCloud: () => void;
}
```

### 6.3 `ActionBar`

职责：

```text
显示当前最重要的动作。
包括推荐升级、可执行 reset、下一主线动作。
```

示例状态：

```text
开局：点击云层 + 购买第一个升级
雨阶可 reset：凝结雨阶成为主按钮
季风可 reset：执行季风循环成为主按钮
风暴可 reset：开启风暴前线成为主按钮
无 reset：显示 1-3 个推荐升级
```

### 6.4 `UpgradeOverlay`

职责：

```text
完整升级列表。
允许内部滚动。
按阶段页签展示。
未解锁系统不显示。
```

可复用当前“升级工作台”实验，但应从常驻右栏变成 overlay。

### 6.5 `ResourceOverlay`

职责：

```text
完整资源列表。
生产者链详情。
每秒收益。
最高 orders。
```

开局不常驻显示。

### 6.6 `FormulaDrawer`

职责：

```text
公式摘要、层级 breakdown、debug orders。
默认隐藏。
只给需要验证数值的人打开。
```

## 7. Reset 展示方式

当前所有 reset 卡片堆在中栏底部，容易裁切和混淆。

建议改为“当前主线行动卡”：

```text
同一时刻只把当前相关 reset 显示在底部行动条。
不可执行的远期 reset 不显示。
即将解锁的 reset 只在目标详情或天气图谱里预告。
```

### 7.1 Reset 优先级

当多个 reset 同时满足时，显示最高优先级：

1. 天空心脏唤醒。
2. 天空心脏脉冲。
3. 气候改写。
4. 风暴前线。
5. 前线回响。
6. 季风循环。
7. 雨阶凝结。

但早期玩家只会看到：

```text
雨阶凝结
季风循环
```

### 7.2 Reset 说明进入详情

主按钮只显示：

```text
凝结雨阶
奖励：雨阶 +1
效果：天气活力收入 ×5
```

复杂说明进入点击后的详情弹层或 tooltip：

```text
会重置哪些内容。
会保留哪些内容。
为什么现在建议 reset。
```

## 8. 交互模式

### 8.1 主屏常驻元素

主屏常驻：

- 顶部 HUD。
- 中央 Mood 舞台。
- 底部行动条。
- 右上角或顶部末端的二级入口。

主屏不常驻：

- 完整资源表。
- 完整公式。
- 全部升级树。
- 全部 reset 卡片。
- 调试 breakdown。

### 8.2 Overlay 类型

建议 overlay 类型：

| Overlay | 用途 | 是否允许滚动 |
|---|---|---|
| 升级全览 | 查看全部已解锁升级 | 是 |
| 资源详情 | 查看资源、生产者、速率 | 是 |
| 天气图谱 | 查看主线阶段、reset 层级 | 是 |
| 公式 | 查看公式和 orders breakdown | 是 |
| 设置 | 暂停、小数显示、重置存档 | 否或少量 |

### 8.3 按钮与图标

当前代码使用 emoji 作为结构图标。后续 UI 重构时建议改成：

```text
CSS/SVG 简单图形
或统一的轻量图标组件
```

如果不引入依赖，可以先用 CSS 形状和文本 badge。不要为了替换图标引入大型 UI 库。

## 9. 响应式方向

### 9.1 桌面

桌面优先使用单屏：

```text
顶部 HUD 高度固定
中央舞台占主要空间
底部行动条高度固定
overlay 浮在舞台上
```

### 9.2 窄屏

窄屏应自然变成：

```text
顶部 HUD
中央舞台
底部行动条
底部导航或菜单按钮
```

不要保留三栏压缩版。

### 9.3 最小可接受范围

周末上线可以优先保证：

```text
桌面 1366x768
桌面 1920x1080
浏览器缩放 100%
```

移动端完整适配可放到 P1，但不能出现明显横向滚动。

## 11. 验收标准

### 11.1 玩家体验验收

```text
开局不会看到云核、气压、风暴胞、气候织线。
玩家一眼能看到当前主目标。
玩家一眼能看到当前最重要按钮。
主屏没有常驻滚动条。
中央舞台是视觉焦点。
进入新阶段时，天空或空岛状态有明显变化。
```

```bash
npm run typecheck
npm run build
npm run simulate:weather-strategies
```

## 12. 需要 GPT Pro 重点评审的问题

请重点批判以下问题：

1. 单屏 HUD 是否比三栏更适合这个游戏的核心循环。
2. 顶部 HUD / 中央舞台 / 底部行动条是否足够承载 1e308 全流程。
3. 阶段化资源显露是否会隐藏必要信息。
4. reset 只显示当前主线动作是否会让玩家失去全局感。
5. Mood 阶段划分是否过细，是否应该合并。
6. Mood 是否应该承担“生态成长”表达，例如岛屿逐渐变绿、云层变厚、天空出现环流。
7. overlay 是否会让升级操作变麻烦。
8. 当前推荐升级是否需要和模拟器策略共用推荐逻辑。
9. 哪些 UI 改动必须在周末上线前完成，哪些可以延后。
10. 如果要用 imagegen 做视觉原型，应该先生成哪些关键画面。
